# Document Type Checklists

Per-document-type focused checklists. Loaded on demand by the skill. If a document spans types, load multiple sections.

The first section ("Universal cross-cutting checks") applies to every document regardless of type.

---

## 0. Universal cross-cutting checks (apply to every document)

- [ ] Document number, title, version, and effective/issue date present and consistent (cover page, header/footer, revision history)
- [ ] Approver / author / reviewer signature blocks complete and dated (or appropriately blank for Early Draft)
- [ ] Revision history populated; latest entry matches the current version
- [ ] No unresolved tracked changes (Final Review and Post-Approval)
- [ ] No unresolved reviewer comments (Final Review and Post-Approval)
- [ ] No placeholder text (TBD, XXXX, [Insert], <XX>) at Final Review or Post-Approval
- [ ] Cross-references resolve to existing controlled documents at the correct version
- [ ] Definitions and abbreviations defined on first use
- [ ] Regulatory references current (no superseded ICH/FDA/EU guidance)
- [ ] References to internal Montai documents use current document IDs and versions
- [ ] Document control metadata complete (owner, controlling SOP, retention period)

---

## 1. SOP / Policy / Work Instruction

- [ ] Purpose / Scope / Applicability stated
- [ ] Roles & Responsibilities matrix complete (RACI-style or equivalent)
- [ ] Procedure section uses imperative voice and is operationally executable
- [ ] Decision points unambiguous
- [ ] Records / forms generated by the procedure are listed and traceable to FRMs or templates
- [ ] Training requirements stated (level, audience, training method)
- [ ] References section lists upstream Policy, parent SOPs, regulatory citations
- [ ] Revision frequency / periodic review interval specified (per Montai's document-control SOP)
- [ ] Exceptions / deviations handling specified or referenced
- [ ] Definitions section covers all GxP-loaded terms used

**Specific to Policy** — should remain principle-level; if a policy contains procedural detail, that detail belongs in an SOP. Flag mismatches.

**Specific to Work Instruction** — must be subordinate to a parent SOP; check the linkage is explicit.

---

## 2. Validation / Qualification / Stability Protocol

- [ ] Objective and acceptance criteria clearly defined
- [ ] Test articles / equipment / materials identified with full traceability (lot numbers, qualification status)
- [ ] Sample sizes and replicate counts justified
- [ ] Statistical treatment specified before data collection (no post-hoc analysis hooks)
- [ ] Pre-defined failure / deviation handling
- [ ] Roles for execution, review, approval, deviation handling
- [ ] Pre-requisites stated (e.g., instrument qualification before method validation)
- [ ] Data capture method specified (paper / electronic) with ALCOA+ controls
- [ ] Reporting plan and report template referenced

**Process validation specifically** — Stage 1/2/3 lifecycle (FDA Process Validation Guidance) addressed; CPP / CQA / CMA terminology consistent with ICH Q8/Q11.

**Stability specifically** — ICH Q1A(R2) zone selection, container/closure, time points, conditions, sampling, statistical evaluation per ICH Q1E.

**CSV protocol specifically** — GAMP 5 categorization stated, risk-based testing depth justified, traceability matrix between user requirements and test cases.

---

## 3. Validation / Qualification / Stability Report

- [ ] Reports against the protocol — every protocol acceptance criterion has an evaluation in the report
- [ ] Deviations from the protocol identified and assessed
- [ ] Raw data references are traceable (notebook page, file path, instrument log)
- [ ] Statistical analysis matches the protocol-specified method
- [ ] Conclusion is supported by the data — and explicitly states whether each acceptance criterion was met
- [ ] Recommendations / next actions stated where applicable
- [ ] All protocol attachments completed (or completed-with-deviation referenced)

**Report-specific Critical patterns**: conclusion that "passes" while data clearly shows failure; selective reporting; missing deviations that the data implies.

---

## 4. Specification

- [ ] All quality attributes covered (identity, assay, impurities, residual solvents, water content, microbiological, physical attributes)
- [ ] Each attribute has a test method reference
- [ ] Each attribute has a numerical acceptance criterion or pass/fail criterion
- [ ] Limits justified — toxicological for impurities (ICH Q3A/B/C/D, M7), pharmacopeial for compendial attributes
- [ ] Limits traceable to the registered IND / IMPD / NDA / MAA dossier
- [ ] Stability-indicating attributes flagged
- [ ] Method/specification version cross-references current

**Drug Substance** — ICH Q6A, Q3A/Q3C/Q3D, M7
**Drug Product** — ICH Q6A/Q6B, Q3B
**Raw materials** — supplier vs. internal specification clarified; release vs. shelf-life criteria distinguished

---

## 5. Master / Batch Records (MBR / BPR / MFR / MPR)

- [ ] Bill of materials matches the registered formulation / process
- [ ] Process steps, equipment, and parameters match the registered process
- [ ] Critical Process Parameters (CPPs) identified with acceptance ranges and operator action limits
- [ ] In-process controls (IPCs) defined with acceptance criteria and decision rules
- [ ] Hold times, hold conditions, and re-process / re-work allowances stated
- [ ] Operator instructions unambiguous; branching steps clearly delineated
- [ ] Yield calculations defined; theoretical, actual, and percent yield captured
- [ ] Sampling plan (in-process, finished) specified
- [ ] Cleaning verification linked to the cleaning validation framework
- [ ] Equipment qualification / cleaning state documented before use
- [ ] Hazardous reagent handling, PPE, waste handling explicit (especially for hazardous reagents like cyanides, halogenated solvents, controlled substances)
- [ ] Investigation triggers defined (out-of-spec, out-of-trend, deviation)
- [ ] Disposition / release decision authority and signature requirements stated

**Critical patterns to watch for**: BPR not aligned with registered process; missing CPP; permissive operator latitude where the registered process is tight; cleaning-validation gap; English/native-language conflicts in translated BPRs (especially Pharmaron Tianjin where Chinese is typically authoritative).

---

## 6. Analytical / Microbiological / Bioanalytical Method

- [ ] Scope and intended use stated (release, stability, in-process, identification, content uniformity, etc.)
- [ ] Principle / theory section technically correct
- [ ] Equipment with required specifications (column dimensions, detector type, resolution, etc.)
- [ ] Reagents at appropriate grade with traceability
- [ ] Reference standards traceable and qualification window stated
- [ ] System suitability criteria appropriate for the intended use
- [ ] Sample / standard preparation steps unambiguous
- [ ] Calculations dimensionally consistent and reproducible
- [ ] Acceptance criteria traceable to the registered specification
- [ ] Reporting format and rounding conventions specified
- [ ] Method qualification / validation status referenced (link to validation report)
- [ ] Safety considerations stated (hazardous reagents, sample handling)

**Critical patterns**: acceptance criteria mismatch with registered spec; method body inconsistent with qualification report; system suitability too permissive; quantitation calculation that produces wrong results with given inputs.

---

## 7. Method Validation / Verification / Transfer Report

- [ ] Validation parameters appropriate for the method category (ICH Q2(R2) Table 1):
  - Identification: specificity
  - Impurity quantitation: specificity, linearity, range, accuracy, precision, LOQ
  - Impurity limit test: specificity, LOD
  - Assay: specificity, linearity, range, accuracy, precision (intermediate precision / robustness)
  - Dissolution / release: specificity, linearity, range, accuracy, precision
- [ ] Each parameter has acceptance criteria, experimental design, results, and pass/fail conclusion
- [ ] Forced degradation / specificity adequately demonstrates indication of stability
- [ ] System suitability criteria carry forward to the routine method
- [ ] Robustness studied (small variations in parameters)
- [ ] Solution stability addressed
- [ ] LOD / LOQ derivation stated (S/N, calibration curve, etc.)
- [ ] Conclusion explicitly maps each acceptance criterion to a pass/fail with cross-reference to data tables

**Verification (USP <1226>)** — confirms a compendial method works for the specific sample matrix; do not over-engineer to full-validation depth.
**Transfer (USP <1224>)** — sending and receiving lab roles defined; protocol-driven; covariance addressed.

---

## 8. Logbooks / Forms / Logs (completed records)

- [ ] All required fields completed
- [ ] Entries contemporaneous (signed and dated as performed)
- [ ] Errors corrected per Montai/site convention (single line through, initial, date, reason)
- [ ] No use of correction fluid / overwriting
- [ ] Each page sequentially used; no skipped pages or gaps
- [ ] Where electronic, audit trail enabled and reviewed
- [ ] Reviewer / second-person verification signatures present where required
- [ ] Anomalies documented with deviation cross-reference

ALCOA+ is the dominant lens for this document type.

---

## 9. Quality Agreement (QTA / QAA)

- [ ] Scope (sites, products, services) explicit and unambiguous
- [ ] Roles and responsibilities matrix covers cGMP elements (release, change control, deviations, complaints, recalls, audit, retain samples, regulatory communications)
- [ ] Authority to release defined (typically client / sponsor for IND-stage)
- [ ] Change-notification timelines defined and meaningful
- [ ] Audit rights specified (frequency, scope, for-cause, regulatory)
- [ ] Subcontracting controls
- [ ] Term, termination, survival of obligations
- [ ] Regulatory inspection notification and support
- [ ] Conflict resolution and dispute handling
- [ ] Specification of which jurisdiction's GMP applies (FDA, EU, ICH, both)
- [ ] FDA Guidance on Contract Manufacturing — Quality Agreements (2016) coverage check
- [ ] EudraLex Vol 4 Chapter 7 coverage check

**Montai-specific**: U.S. IND obligations should be explicit (this was a focus of the QTA.0283 redline for MTAI-1025 / Aptuit-Evotec). KPI reporting expectations stated.

---

## 10. Certificate of Analysis (CoA / CoC)

- [ ] Material identified (name, code, lot, manufacture date, expiry / re-test date)
- [ ] Specification reference cited at the correct version
- [ ] Each specification attribute has a result and pass/fail call
- [ ] Test methods cited with versions
- [ ] Out-of-specification entries (none expected on a release CoA) — if present, deviation reference required
- [ ] Authorized signatory (QC, QA, QP as applicable)
- [ ] Storage and shipping conditions referenced
- [ ] Container/closure described
- [ ] Conformance statement (e.g., "Material conforms to specification SPE-XXXXX rev YY")

---

## 11. Change Control

- [ ] Change description, scope, classification (minor / moderate / major / Type I/II)
- [ ] Justification / rationale for the change
- [ ] Risk assessment (per ICH Q9(R1)) — appropriate tool used (FMEA, risk matrix)
- [ ] Impact assessment across affected systems (validation, registration, labeling, suppliers, training, documentation)
- [ ] Action plan with owners, due dates, dependencies
- [ ] Effectiveness check / verification method
- [ ] Approvals from required functional areas
- [ ] Closure criteria explicit

---

## 12. CAPA

- [ ] Source / trigger documented (deviation, complaint, audit finding, trend, periodic review)
- [ ] Problem statement clear and specific (what, when, where, scope)
- [ ] Root cause analysis tool appropriate (5-whys, fishbone, fault tree); root cause stated, not just contributing factors
- [ ] Corrective actions address the immediate issue
- [ ] Preventive actions address recurrence at a system level
- [ ] Effectiveness check / verification plan with measurable criteria
- [ ] Owners and due dates with clear accountability
- [ ] Linkage to Montai SOP-04 (CAPA Management) explicit
- [ ] Closure approval matches the SOP-defined approval matrix

---

## 13. Deviation / Out-of-Specification (OOS) Record

- [ ] Description specific (what, when, where, who, batch/sample affected)
- [ ] Investigation plan covers all credible root causes
- [ ] Phase 1 (laboratory) for OOS per FDA OOS Guidance — clear and complete
- [ ] Phase 2 (manufacturing/scientific) where Phase 1 doesn't identify cause
- [ ] Impact assessment on batch / product / process / other batches in scope
- [ ] Disposition decision supported by investigation
- [ ] Corrective action and preventive linkage (CAPA initiation if warranted)
- [ ] QA approval and timeline for closure within SOP-defined limits

---

## 14. Audit Report (internal, vendor, regulatory mock)

- [ ] Audit scope, duration, audit team, and auditees defined
- [ ] Methodology stated (document review, interviews, walk-through, observation)
- [ ] Findings categorized (Critical / Major / Minor / Observation) with the same rubric as document reviews
- [ ] Each finding has objective evidence (document reference, observation, interview note)
- [ ] Each finding has a regulatory or SOP reference
- [ ] Audit response process and timeline stated
- [ ] Distribution list and confidentiality controls

---

## 15. Computerized System Validation (CSV) records

- [ ] User Requirements Specification (URS), Functional Specification (FS), Configuration Specification (CS), Design Specification (DS) appropriate for system category (GAMP 5)
- [ ] Risk assessment driving testing depth
- [ ] IQ / OQ / PQ test cases trace to requirements via traceability matrix
- [ ] User acceptance and migration plans
- [ ] Audit trail validated
- [ ] User access matrix and segregation of duties
- [ ] Periodic review schedule defined
- [ ] Backup, restore, business continuity plan tested
- [ ] Decommissioning and data retention plan

**Cross-references**: GAMP 5, EU Annex 11, 21 CFR Part 11, PIC/S PI 011, PI 041.

---

## 16. Clinical documents (GCP)

### Clinical protocol (ICH E6(R3), E8(R1), E9(R1))

- [ ] Background and rationale
- [ ] Objectives and endpoints (primary, secondary, exploratory) with estimands per E9(R1)
- [ ] Study design (randomization, blinding, comparator)
- [ ] Eligibility criteria (inclusion / exclusion)
- [ ] Treatment and dosing
- [ ] Safety monitoring and stopping rules
- [ ] Data Management and Statistical Analysis Plan reference
- [ ] Ethics, informed consent, regulatory framework
- [ ] Adverse event definitions and reporting per ICH E2A
- [ ] Trial registration commitment

### Clinical Study Report (ICH E3)

- [ ] Structure follows E3 sections
- [ ] Conclusions trace to data; benefit-risk discussed
- [ ] Major protocol deviations summarized with impact
- [ ] Safety reporting completeness checked against site-level data

### Investigator Brochure

- [ ] Up-to-date safety information
- [ ] Pharmacology / toxicology summary
- [ ] Reference safety information (RSI) clearly identified

### Site monitoring report

- [ ] Monitoring frequency per the monitoring plan
- [ ] Source data verification scope and findings
- [ ] Protocol deviation tracking and follow-up
- [ ] Action items with owners

---

## 17. GLP study documents (21 CFR Part 58, OECD GLP)

- [ ] Study protocol approved before study initiation
- [ ] Study director identified and signature on protocol
- [ ] Test article characterization
- [ ] Study schedule, test system, controls
- [ ] Data capture, archiving, retention
- [ ] QA Unit oversight and inspections documented
- [ ] Final study report includes the QAU statement
